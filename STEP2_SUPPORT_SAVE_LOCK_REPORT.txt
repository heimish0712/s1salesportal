STEP2 - 영업지원요청 저장 안정화

기준 파일:
- 영업전산포탈_STEP1_자료발송이벤트정리.zip

수정 파일:
- 09_SupportRequestService.gs

주요 반영 내용:
1) savePortalSupportRequest(payload)를 Lock wrapper + core 함수 구조로 분리
   - savePortalSupportRequest(payload)
   - runPortalSupportWriteLockedP210_(label, callback)
   - savePortalSupportRequestCoreP210_(payload)

2) 영업지원요청 저장 핵심 구간을 Script Lock으로 보호
   - clientRequestId 중복 확인
   - 신규/수정 rowNo 결정
   - 첫 빈 행 탐색
   - 접수번호 발급
   - 권한에 따른 완료/처리 필드 보정
   - 행 저장
   - 신규 요청의 마스터 메모 반영
   - 영업지원 캐시 bust

3) 컬럼별 setValue 반복 제거
   - 기존: PORTAL_CONFIG.SUPPORT_HEADERS.forEach(...) 내부에서 셀마다 setValue()
   - 변경: 행 값을 구성한 뒤 setValues()로 연속 컬럼 블록 단위 저장
   - 일반 헤더 구조(1~19열 연속)에서는 1회 setValues()로 저장됨

4) 활동로그 기록을 Lock 밖으로 이동
   - 접수/저장 자체는 Lock 안에서 완료
   - 활동로그 실패가 영업지원요청 저장 성공을 막지 않도록 분리

5) 신규 헬퍼 추가
   - ensurePortalSupportHeadersForWriteP210_()
   - ensurePortalSupportRowExistsP210_()
   - getPortalSupportWriteLastColP210_()
   - writePortalSupportRowAtomicP210_()
   - groupConsecutivePortalSupportColsP210_()

검증:
- 09_SupportRequestService.gs Node 문법 검사 통과
- 09_SupportRequestService.gs top-level function 중복 없음
- 기존 영업지원 저장부의 셀 단위 setValue 반복 제거 확인
- savePortalSupportRequest 클라이언트 호출명 유지 확인

주의:
- 실제 Apps Script/Spreadsheet 동시저장 실부하 테스트는 배포 환경에서 별도 확인 필요
