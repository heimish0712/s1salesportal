STEP7 - 브라우저 고객 캐시 용량 정리/계정 분리 반영 보고서

기준 산출본
- 영업전산포탈_STEP6_권한조회캐시추가.zip

수정 파일
- Client_State.html
- Component_CustomerSearch.html

목표
- 고객 상세 localStorage 캐시가 계속 누적되어 브라우저 저장공간을 압박하는 문제 방지
- 고객 검색 페이지 sessionStorage 캐시가 과도하게 쌓이는 문제 방지
- 같은 브라우저에서 계정이 바뀌었을 때 이전 계정의 고객 캐시가 섞이는 문제 방지
- localStorage quota 초과 시 조용히 저장 실패하는 기존 구조를 줄이고, 정리 후 재시도하도록 보강

반영 내용
1. 고객 검색 페이지 캐시 정리
- sessionStorage의 portalCustomerPage:* 캐시는 최대 70개까지만 유지
- 10분 초과 캐시는 조회/정리 시 삭제
- 현재 권한 이메일과 다른 permissionEmail을 가진 페이지 캐시는 즉시 삭제
- 메모리 페이지 캐시도 동일하게 70개 기준으로 오래된 항목부터 삭제

2. 고객 상세 localStorage 캐시 정리
- salesPortal.customerDetail.v58.masterBackedOnly.* 캐시는 물리 key 기준 최대 260개 유지
- v45 구버전 상세 캐시도 정리 대상에 포함
- CUSTOMER_DETAIL_STORAGE_MAX_AGE_MS 기준 2시간 초과 캐시는 삭제
- __lastAccessedAt 기준 LRU 방식으로 오래된 항목부터 제거
- 저장공간 초과가 나면 상세 캐시를 강제 정리한 뒤 localStorage.setItem을 1회 재시도

3. 고객 상세 메모리 캐시 정리
- customerDetailInlineCache는 최대 420개 key만 유지
- CNO/ROW 중복 key가 존재하는 현재 구조를 감안해 물리 key 기준으로 제한
- 오래 접근하지 않은 key부터 삭제

4. 계정 변경 시 고객 캐시 분리
- localStorage에 salesPortal.customerCache.owner.v1 owner marker 추가
- 권한 로딩 성공 후 현재 이메일과 owner marker를 비교
- owner가 없거나 이전 owner와 다르면 고객 관련 캐시를 정리
  * 고객 검색 페이지 캐시
  * 고객 검색 인덱스 localStorage
  * 고객 상세 localStorage
  * 고객 상세 메모리 캐시
- 권한 확인 전에는 localStorage 고객 검색 인덱스를 사용하지 않도록 보강

5. 검색 인덱스 localStorage owner 정보 추가
- CUSTOMER_INDEX_STORAGE_KEY 저장 payload에 cacheOwner 추가
- 로딩 시 permissionEmail/cacheOwner가 현재 계정과 다르면 인덱스 캐시 삭제 후 사용하지 않음

검증
- 모든 .gs 문법 검사 통과
- 모든 .html <script> 문법 검사 통과
- 변경 파일 top-level function 중복 0건
- 전체 top-level function 중복 0건
- STEP7_STATIC_CHECK.json 포함

한계
- 브라우저 캐시 정책만 정리한 단계이며, 서버 CacheService/ScriptCache 용량 정책은 이번 범위에 포함하지 않음
- localStorage의 정확한 byte 사용량을 계산하지 않고, entry 개수와 TTL 중심으로 제한함
- 같은 고객 상세가 CNO/ROW 2개 key로 저장되는 구조는 호환성 때문에 유지함

운영 효과
- 고객을 많이 열어볼수록 localStorage가 쌓이는 문제 완화
- 브라우저 quota 초과 시 상세 캐시 저장 실패 가능성 감소
- 같은 PC/브라우저에서 다른 계정으로 접속했을 때 이전 고객 캐시가 섞일 가능성 감소
