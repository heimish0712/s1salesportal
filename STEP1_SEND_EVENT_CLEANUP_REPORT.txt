[STEP1] 자료발송(Component_Send.html) 이벤트 충돌 정리 보고

수정 목적
- 자료발송 화면에 남아 있던 구버전 단일 작업창 이벤트 함수와 신버전 다중 작업창 이벤트 함수의 동일 함수명 중복을 제거.
- 최종 사용 흐름은 P2-11/P2-13 다중 작업창 구조 기준으로 유지.
- 검색 응답 순서 역전으로 오래된 검색 결과가 최신 화면을 덮는 문제를 방지.

주요 수정
1. Component_Send.html 내 중복 top-level 함수 제거
   - renderSendPage
   - restoreSendPageStateV50_
   - resetSendPageAllV50
   - searchSendCustomers
   - searchSendCustomersPage
   - renderSendResults
   - selectSendCustomer
   - renderInlineMailProgress_
   - openMailProgressModal
   - closeMailProgressModal
   - updateMailProgressUi
   - pollMailProgress
   - finishMailProgressModal
   - cancelCurrentMailSend
   - setSendMessage
   - setSendMessageHtmlV75_
   - finishPortalReviewProgressV75_
   - preparePortalMailFilesForReviewFromUi
   - sendPortalMailFromUi
   - openSendForCustomer

2. 유지한 최종 기준
   - 다중 발송 작업창 기반 함수만 유지
   - 탭별 selectedDetail / selectedFileKeys / 수신자 / 참조 / reviewSession / mailRunId / progress 상태 유지
   - 완료/실패 탭 정리, 중복 고객 탭 이동, 새로고침 복구 로직 유지

3. 검색 이벤트 순서 안정화
   - workspace별 searchSeq 추가
   - 같은 탭에서 검색어를 빠르게 바꿨을 때, 먼저 돌아온 오래된 응답이 최신 검색 결과를 덮지 않도록 방지

검증
- Component_Send.html top-level function 중복 0건 확인
- Component_Send.html JavaScript 문법 검사 통과(node --check)
- 자료발송 버튼에서 호출하는 주요 함수 유지 확인
  - preparePortalMailFilesForReviewFromUi
  - sendPortalMailFromUi
  - searchSendCustomers
  - selectSendCustomer
  - openSendForCustomer

주의
- 이번 단계는 프론트 자료발송 이벤트 충돌 정리입니다.
- 실제 Gmail/Hiworks 발송 성공 여부는 Apps Script 배포 후 테스트 발송으로 확인해야 합니다.
- 다음 단계 권장: 영업지원요청 저장 Lock 적용 및 setValues 일괄 저장화.
