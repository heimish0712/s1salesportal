STEP21 검색/로딩 튐 방지 + 메모 전체표시 보강 보고서

기준선: STEP20_중점고객관리리스트표시_20개페이징수정

문제 원인
1. 고객 상세 검색에서 searchCustomerPage()가 실행될 때마다 initializeCustomerSearchIndex()가 서버 getCustomerSearchIndexData()를 다시 호출했다.
   - 브라우저에 이미 검색인덱스가 있어도 매 검색/페이지 이동 때 서버 인덱스 로딩이 병렬로 돌았다.
   - 이 서버 응답의 success handler가 현재 활성 검색 화면을 다시 renderResults()로 갱신하면서 결과가 중간에 튀어 보였다.
   - 사용자 입장에서는 구글시트를 가볍게 보는 것이 아니라 검색 중간에 데이터가 움직이는 것처럼 보였다.

2. 검색 시작 시 resultArea를 무조건 loading HTML로 갈아끼웠다.
   - 실제로는 localStorage/메모리 인덱스에서 즉시 결과를 만들 수 있는데도 화면을 먼저 비웠다.
   - 캐시가 있는 검색에서도 깜빡임이 발생했다.

3. 중점고객관리 화면은 응답이 올 때마다 toolbar 전체를 다시 렌더링했다.
   - 오래된 서버 응답을 무시하는 request sequence가 없었다.
   - 필터/페이지 요청이 빠르게 겹치면 늦게 온 응답이 최신 화면을 덮을 수 있었다.

4. 메모는 검색인덱스_DB의 '메모요약'을 먼저 사용했다.
   - 목록 행의 메모 요약은 짤려도 되지만, 상세/펼침/수정 textarea에서는 절대 짤린 값을 쓰면 안 된다.
   - 기존 구조는 full detail 도착 전 summary detail이 메모요약을 들고 있어, 상세/펼침/수정에서 짧은 메모가 잠깐 보이거나 저장될 위험이 있었다.

수정 내용
1. 고객 검색 일반 동작은 브라우저 인덱스/페이지 캐시 우선으로 변경했다.
   - 이미 ready 상태의 검색인덱스가 있으면 일반 검색/페이지 이동에서 서버 인덱스를 다시 당기지 않는다.
   - 서버 인덱스 갱신은 최초 진입, 명시 refresh, 외부변경 감지, 인덱스 미보유/오래됨 상황에만 실행된다.

2. 검색 시작 시 빠른 표시 가능 여부를 먼저 확인한다.
   - local index 또는 page cache로 즉시 표시 가능한 경우 resultArea를 loading으로 비우지 않는다.
   - 캐시/인덱스가 없을 때만 loading을 표시한다.

3. 검색인덱스 서버 응답이 자동으로 화면을 재렌더링하지 않게 했다.
   - initializeCustomerSearchIndex()는 기본적으로 인덱스 데이터만 갱신한다.
   - renderAfterLoad:true를 명시한 경우에만 현재 검색 화면을 렌더링한다.
   - 기존 검색 결과가 서버 인덱스 응답 때문에 갑자기 바뀌는 문제를 줄였다.

4. 중점고객관리 화면 안정화
   - requestSeq 추가: 오래된 응답은 화면 반영하지 않는다.
   - client memory cache 추가: 같은 조건/페이지는 서버 호출 없이 즉시 표시한다.
   - 응답 시 toolbar 전체 재생성을 피하고, summary/today/list 영역만 갱신한다.
   - 새로고침 버튼은 client cache와 server cache를 모두 비우고 재조회한다.

5. 메모 전체표시 보강
   - buildCustomerDetailFromRowSummary_()에서 검색인덱스 메모요약을 상세/수정용 memo로 쓰지 않는다.
   - 펼침 패널은 master full detail이 도착하기 전까지 memo textarea를 readonly + loading 상태로 둔다.
   - memo inline edit은 full detail을 먼저 가져온 뒤 textarea를 연다.
   - full detail이 없는 상태에서 요약 메모를 수정/저장하지 못하게 막았다.
