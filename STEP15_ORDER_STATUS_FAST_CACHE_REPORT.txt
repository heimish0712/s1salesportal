STEP15 발주여부 조회 속도/캐싱 개선 보고서

기준선: STEP14_발주여부한줄표시수정

요청사항
- 고객 상세검색 펼침 영역의 발주여부가 다른 상세 데이터보다 늦게 표시되는 문제 개선.
- 발주여부도 고객 상세/프리패치/브라우저 캐싱 흐름에 같이 태우기.

반영 파일
- 05_CustomerService.gs
- 16_ContractCompleteService.gs
- Component_CustomerSearch.html

핵심 반영
1. 수주확정/계약완료 시트 전체 row를 매번 순회하지 않도록 compact order lookup 캐시 추가.
   - CacheService key: PORTAL_CUSTOMER_ORDER_LOOKUP_P260
   - 고객번호/회사명 -> 계약번호/행번호 맵으로 축약 저장
   - 기존 계약완료 목록 캐시와 동일 TTL(180초) 사용
   - clearContractCompleteCacheV69_() 호출 시 lookup 캐시도 같이 삭제

2. getPortalCustomerOrderInfoByTargetP250_()를 lookup 우선 조회 방식으로 변경.
   - 기존 공개/내부 함수명은 유지
   - 기존 listContractCompleteRowsV69() fallback 유지

3. getCustomerDetailsBatch()에서 발주여부를 batch 상세조회 결과에 포함.
   - batch당 order lookup 1회 준비
   - 각 detail에 orderInfo 포함
   - 프리패치 결과가 고객 상세 캐시/localStorage 캐시에 저장되므로 재진입 시 빠르게 표시

4. buildCustomerDetailFromObj_()에서 lite 상세에도 orderInfo 포함 가능하도록 변경.
   - 기존 single full detail 흐름 유지
   - includeOrderInfo:false 옵션은 계속 존중

5. 클라이언트 즉시 표시 보강.
   - 요약/인덱스 detail만 있는 순간에도 진행현황 기준으로 즉시 O/X 표시
   - 서버 상세 보강 전에도 `확인중`에 오래 머물지 않도록 함
   - 실제 중복 발주 여부는 서버 발주 처리 함수에서 다시 검증하므로 안전장치 유지

보존한 것
- 발주하기 버튼 로직 변경 없음
- 수주확정/계약완료 이관 로직 변경 없음
- 기존 계약번호 중복 방지 유지
- 고객 상세검색/자료발송/영업지원/오늘할일 업무 흐름 변경 없음

검증
- 모든 .gs 문법 검사 통과
- 모든 .html script 문법 검사 통과
- 변경 파일 내부 신규 중복 함수 없음
- 기존 주요 함수 존재 확인
